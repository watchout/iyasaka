## 1. 企業アイデンティティ

### 1.1 ミッション

**「今ある『不』を、未来の『光』へ。」**

### 1.2 社名の意味

「弥栄（いやさか）」= いよいよ栄える。関わる全ての人・地域が共に繁栄する循環を実装するチーム。

### 1.3 コア・バリュー

| バリュー | 原則 | 設計への影響 |
|---------|------|------------|
| 現場第一主義 | 綺麗なコードやスペックより、現場の「溜息」を消すことを優先 | UI/UXは現場で使える実用性を最優先 |
| プラスの精神 | 既存の努力を否定せず、技術を「プラス」して価値を最大化 | 既存ワークフローを破壊しない設計 |
| 弥栄の循環 | 一社勝ちではなく、顧客・利用者・地域が等しく潤う仕組み | JV/サブスク型リスク分担モデル |

## 2. ナラティブOS（意思決定の根幹）

### 2.1 3つの変換動詞

全製品・全コミュニケーションは以下の3つのフェーズに分類される。

| 動詞 | 英語 | カラー | 対象製品 | 不（Before） | 光（After） |
|------|------|--------|---------|-------------|------------|
| 整える | Foundation | #3182ce | 弱電プラス、ミエルプラス、dev-OS | 不透明、不安、属人化 | 確信、可視化、安心 |
| つなぐ | Expansion | #319795 | 配信プラス、多言語プラス | 分断、孤立、伝わらない | 接続、共鳴、無境界 |
| 育てる | Evolution | #38a169 | OmotenasuAI、AIプラス | 形骸化、余裕の欠如 | 循環、共創、持続可能性 |

### 2.2 ナラティブ・ループ構造

全てのページ・セクションは以下の物語構造に従う:

```
不（Pain） -> 原因（Cause） -> 仕組み（Solution） -> 光（Light） -> 弥栄（Iyasaka）
```

**設計原則**: この構造に従わないコピー・UI構成は修正対象。

## 3. 製品ポートフォリオ

### 3.1 製品一覧と優先度

| 順位 | 製品 | p_id | 種別 | ターゲット業種 |
|------|------|------|------|--------------|
| 1 | ミエルプラス | mieru-plus | パッケージSaaS | 建設・製造・物流 |
| 2 | 弱電プラス | jakuden-plus | サブスク保守 | 店舗・オフィス全般 |
| 3 | OmotenasuAI | omotenasu-ai | 業種特化AI | ホテル・旅館 |
| 4 | AIプラス | ai-plus | 受託AI開発 | 全業種 |
| 5 | 配信プラス | haishin-plus | JV型サービス | 貸会議室・イベント |
| - | dev-OS | - | 技術基盤 | （製品として訴求しない） |

### 3.2 共通テーマ

全製品の深層ペインの共通項は **「経営者の孤独」**:
- 誰にも相談できない
- 誰も代わってくれない
- 専門外のことは判断できない
- 言い値で払うしかない

**IYASAKAの価値**: 「その孤独を、私たちが引き受ける」

### 3.3 深層ペイン一覧（設計判断の根拠）

| 製品 | ペイン1 | ペイン2 |
|------|---------|---------|
| OmotenasuAI | ホスピタリティの限界に対する罪悪感 | 「人」に依存しすぎるリスクの恐怖 |
| 弱電プラス | 「繋がらない・鳴らない」時の絶望感 | ブラックボックス化する設備費用 |
| 配信プラス | 「場所貸し」という付加価値の死 | クリエイターの「作業者」化 |
| ミエルプラス | 「進捗の嘘」への不信 | 属人化した「脳内データ」の喪失 |
| AIプラス | 「うちは特殊だから」という閉塞感 | 外注への不安（品質 vs 予算） |

**利用ルール**: LP・広告コピーはこの実体験ベースのペイン情報を根拠に作成する。AIが想像で作成したペインは使用しない。

## 5. ビジネスモデル原則

### 5.1 リスク分担型モデル

IYASAKAは「高額一括投資」を求めない。

| モデル | 適用製品 | 顧客メリット |
|-------|---------|------------|
| サブスクリプション | 弱電プラス、ミエルプラス | 月額固定で予算管理しやすい |
| JV（共同事業） | 配信プラス | 初期投資なし、レベニューシェア |
| 受託開発 | AIプラス | 段階的な投資が可能 |

### 5.2 CTA規格

全製品のCTAは以下の形式に統一:
```
https://iyasaka.co/contact?p_id=[プロダクトID]
```

製品LPからの遷移は診断経由を推奨:
```
LP -> 診断フォーム -> 結果画面 -> 無料相談CTA
```

## 7. コード/UIにおける禁止事項

| 禁止 | 理由 |
|------|------|
| コードやコメントに絵文字 | プロダクトUI以外では使用しない |
| console.log を本番に残す | デバッグ用途のみ |
| 農業を現行サービスに含める | 「未来の展望」としてVisionセクションでのみ言及 |
| 仕様書にない機能を勝手に実装 | AI中断プロトコルに従い確認を求める |
| 想像で作成したペインでコピーを書く | PRODUCT_PAIN_POINTS.md の実体験ベースを使用 |

## 8. 戦略的意思決定ルール

フィードバックが衝突した場合の優先順位:

1. **CVR**: コンバージョン率が落ちる可能性のある演出は後置
2. **FV**: ファーストビューは具体的痛み訴求、2段目でナラティブ回収
3. **速度**: LCP/INP/CLSを最優先のガードレール
4. **判断保留**: 割れたらA/Bテスト前提で両案を残す
