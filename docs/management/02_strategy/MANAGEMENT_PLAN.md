# 経営計画（3〜6ヶ月）

## 0. この計画の目的

- 3〜6ヶ月で「作って回す」体制を完成させる
- 3ヶ月以内にリリースし、プレローンチで **無料先行予約** を獲得する
- 「闇雲」ではなく、数字で分解し、週次で実行を最大化する

## 1. 期間と目標（数値）

### 3ヶ月（12週間）

- 北極星: **ホテルAIの無料先行予約 3件（合計）**
- 補助: 予約→商談→有効リード→リード→流入をKPIツリーで分解して追う  
  - `../03_kpi/KPI_TREE.md` / `../03_kpi/KPI_DICTIONARY.md`

### 6ヶ月

- 目標: 「予約が偶然ではなく再現性で出る」状態（週次WBRで改善が回る）
- 補助: 先行導入/先行設計の実績（ケース）を作り、提案の型（SOP）を固定する

## 1.5 数値計画（1年）

- 目標（1年）: **経常利益 3,000万円**
- 前提（暫定）: 役員報酬 100万/月、社員2名×40万/月、粗利率 50%
- 広告/固定費の前提: サーバー/AI 15〜20万円/月、広告 50〜100万円/月
- ワークシート: `FINANCIAL_PLAN_1Y.md`

## 2. 前提（いまの現実）

- ほぼすべてのプロダクトは開発中
- だから、**個別プロダクトの完成度より、全体フローの完成を優先**する
- 入口は統一する（診断/相談）  
  - 入口で製品を売り分けず、提案で分岐させる

### 2.1 主戦場（3〜6ヶ月）

- 主戦場: **ホテルAI（客室AIコンシェルジュ / ホテルPMS / ホテルCRM）**
- 入口（楔）: **客室AIコンシェルジュ**
- 初期ターゲット: **小規模ホテル（〜50室目安）**
- 位置づけ: 挑戦（新規の勝ち筋を作る）
- 補助（キャッシュ/信用の基盤）: 弱電事業（成長）

### 2.2 ポートフォリオ運用ルール（12週間）

主戦場を「やり切る」ため、12週間は運用ルールを固定する。

- 時間配分（原則）: **ホテルAI 80% / 弱電 20% / その他 0%（新規の獲得活動はしない）**
- 例外（許可する“その他”）:
  - 既存の約束/関係維持（例: 配信JV先との打ち合わせ）
  - ホテルAIの前進に直結する開発（例: 客室AIコンシェルジュの導入に必要な連携）
  - 小さな検証（例: AIプラスのテストマーケ）※週次の「実験」枠で扱う

#### 他製品の扱い（12週間）

- 弱電事業: **キャッシュ＋ホテルAI入口**（ホテル現場の信頼/導線を作る）
- 配信プラス: **関係維持のみ**（JV先との打ち合わせは継続、新規獲得は追わない）
- ミエルプラス: **開発は可**（ただしホテルAIの主戦場を崩さない範囲。新規獲得は追わない）
- AIプラス: **テストマーケのみ**（受託は後。12週間は“検証”として扱う）
- dev-OS: **内部基盤のみ**（品質/速度を上げる。外販はしない）

## 3. 戦略（経営の意思決定）

### 3.1 優先順位

1. 計測できる状態（KPI定義・パイプライン定義・記録）が先
2. 予約を生むチャネル（紹介/提携/アウトバウンド）を先
3. サイト/コンテンツは「信頼形成 + 刈り取り」を担う

### 3.2 意思決定ルール（毎週固定）

- 今週の重点は **最大3つ**（増やさない）
- KPIが動かない週は、行動KPI（接触/打診/フォロー/実験/出力）を増やす
- 定義が揺れたら `../03_kpi/KPI_DICTIONARY.md` を更新して統一する

※鬼速PDCAの「課題を3つに絞る」考え方を採用する  
参考: `https://tomitazuu.com/archives/537` / `https://hr-mitas.jp/blogs/okr-performance-management/53/`

## 7. リスクと対策

- 製品が多くて説明が散る
  - 対策: 入口を統一し、提案で分岐する（予約の型を固定）
- 改善が遅い（議論が長い）
  - 対策: 週次で最大3施策、定義は辞書で固定
- 「研究」に逃げる
  - 対策: 研究テーマは週1つ、必ずSOP/KPIに変換できたものだけ採用するのだけ採用する
